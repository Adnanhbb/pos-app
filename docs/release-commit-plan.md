This document prepares the release commit/tag plan after production build verification and git readiness audit. It is planning only: no commit, tag, push, deploy, runtime behavior change, or auto-sync enablement is performed by this document.

## Commit-Worthy Categories

Commit these categories after review:

- `api/` backend PHP source and schema updates
- `src/` frontend source updates
- `scripts/` developer/test/diagnostic/release tooling
- `docs/` architecture, checkpoint, checklist, and release planning documents
- `.env.example` and `.env.production.example` templates with placeholders only
- `package.json` for npm script additions
- `package-lock.json` if dependency or npm metadata changes are intentional

Do not commit real secrets, production DB credentials, raw tokens, payload exports, or generated local backup data.

## Ignored / Generated Categories

These should stay ignored and normally should not be committed:

- `dist/` production build output
- `backups/` local IndexedDB/MySQL backup JSON exports
- `releases/` generated release manifest JSON files unless a specific manifest is intentionally archived outside normal source history
- `tsconfig.tsbuildinfo`
- `node_modules/`
- `*.log`, including `debug.log`
- local `.env` files covered by `*.local` / environment-specific local config rules

## Recommended Commit Grouping

Because the current working tree is large, prefer staged commits by architecture area rather than one opaque commit if time allows:

1. Low-risk CRUD sync foundation and tests.
2. Backend transaction replay chain and tests.
3. Auth/session and CRUD auth audit/enforcement foundation.
4. Hydration/reconciliation/auto-sync gate tooling.
5. Backup/export/validation/release tooling.
6. Developer Control Panel foundation.
7. Production deployment/operational documentation and checkpoints.

If a single checkpoint commit is required for handoff, use the suggested commit message below and verify the staged diff carefully.

## Final Recommended Staging Commands

Run these from the repository root after reviewing the working tree:

```powershell
git status

git add .gitignore .env.example .env.production.example package.json package-lock.json
git add api
git add src
git add scripts
git add docs
git add -u tsconfig.tsbuildinfo

```

The grouped staging intent is:

- source/runtime files: `src/`
- backend/API files: `api/`
- developer, test, backup, release, and diagnostics scripts: `scripts/`
- architecture, checkpoint, deployment, backup, and release documents: `docs/`

- package files: `package.json`, `package-lock.json`
- generated build-info tracking cleanup: `git add -u tsconfig.tsbuildinfo` stages the index-only removal while leaving the local ignored file on disk
